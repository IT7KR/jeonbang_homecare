## 1. 개요

### 1.1 프로젝트 배경

전방 홈케어는 양평/가평 지역을 중심으로 홈케어 서비스(제초, 정원관리, 외벽청소, 간단수리 등)를 제공하는 사업체입니다. 현재 오프라인 및 전화 기반으로 운영 중인 서비스 신청 프로세스를 디지털화하여 마케팅 트래픽을 효율적으로 처리하고, 고객-관리자-협력사 간의 연결을 체계화하고자 합니다.

### 1.2 프로젝트 목적

- **고객 접점 디지털화**: 온라인을 통한 서비스 신청 채널 구축
- **업무 효율화**: 신청 접수, 일정 관리, 파트너 배정 프로세스 단순화
- **마케팅 연계**: 마케팅 캠페인으로 유입된 트래픽의 효과적인 전환
- **확장 기반 마련**: 향후 지역 확장 및 기능 고도화를 위한 기반 구축

### 1.3 프로젝트 범위

#### 1.3.1 범위 내 (In Scope)

| 구분         | 항목                                                              |
| ------------ | ----------------------------------------------------------------- |
| Front Office | 고객용 메인페이지, 서비스 신청 페이지, 파트너 등록 페이지         |
| Back Office  | 고객 신청 관리, 파트너 승인 관리, 일정/방문 모니터링, 시스템 설정 |
| 외부 연동    | SMS 발송 (알리고 API)                                             |
| 인프라       | AWS Lightsail 기반 호스팅, SSL 인증서                             |

#### 1.3.2 범위 외 (Out of Scope)

| 항목                 | 비고                                    |
| -------------------- | --------------------------------------- |
| 실시간 채팅          | 운영 부담으로 제외                      |
| 온라인 결제          | 현장 결제로 운영                        |
| 전자계약             | 외부 서비스 이용 권장                   |
| 고객 로그인/회원가입 | 비회원 신청만 지원, 관리자가 SMS로 안내 |
| 이메일 알림          | SMS만 사용                              |

### 1.4 용어 정의

| 용어   | 정의                                                 |
| ------ | ---------------------------------------------------- |
| 고객   | 홈케어 서비스를 신청하는 일반 사용자                 |
| 파트너 | 전방 홈케어와 협력하여 서비스를 제공하는 업체/개인   |
| 관리자 | 시스템을 운영하고 신청/파트너를 관리하는 내부 담당자 |
| 신청   | 고객이 제출한 서비스 요청 건                         |
| 배정   | 특정 신청 건을 파트너에게 할당하는 행위              |

## 3. 사용자 역할 및 페르소나

### 3.1 역할 정의

#### 3.1.1 고객 (Customer)

**역할 설명**: 홈케어 서비스가 필요한 양평/가평 지역(및 인근) 거주자 또는 별장/세컨하우스 소유자

**주요 행동**:

- 서비스 정보 탐색 및 확인
- 서비스 신청서 작성 및 제출
- 전화/문자를 통한 문의

**권한**:

- 메인페이지 및 서비스 소개 열람
- 서비스 신청 폼 작성 및 제출
- 파트너 등록 페이지 열람 (정보 확인 목적)

**제한사항**:

- 로그인 기능 없음
- 신청 내역 조회 불가 (SMS로 안내 받음)
- 관리자 페이지 접근 불가

#### 3.1.2 파트너 (Partner)

**역할 설명**: 전방 홈케어와 협력하여 실제 서비스를 제공하는 업체 또는 개인 사업자

**주요 행동**:

- 파트너 등록 요청서 작성 및 제출
- 사업자등록증 등 필요 서류 업로드

**권한**:

- 파트너 등록 페이지 접근 및 폼 제출

**제한사항**:

- 별도 파트너 포털 없음 (현 버전)
- 등록 후 진행 상황은 전화/SMS로 안내 받음
- 직접 신청 건 조회 불가

#### 3.1.3 관리자 (Admin)

**역할 설명**: 전방 홈케어 내부 운영 담당자

**주요 행동**:

- 고객 신청 접수 확인 및 처리
- 파트너 등록 요청 검토 및 승인/반려
- 신청 건 파트너 배정
- 일정 및 방문 현황 모니터링
- 고객에게 SMS 발송
- 시스템 설정 관리

**권한**:

- Back Office 전체 접근
- 모든 데이터 CRUD
- SMS 발송
- QR 코드 생성
- 시스템 설정 변경

### 9.6 반응형 처리 가이드

```tsx
// 그리드 반응형 예시
```

### 9.7 모듈화 원칙

#### 컴포넌트 설계 원칙

1. **Single Responsibility**: 하나의 컴포넌트는 하나의 역할
2. **Props Interface**: 명확한 Props 타입 정의
3. **Composition**: 작은 컴포넌트 조합으로 복잡한 UI 구성
4. **Controlled/Uncontrolled**: 폼 요소는 React Hook Form으로 통일

#### 공통 훅 (Hooks)

```
hooks/
├── useApplications.ts     # 신청 데이터 조회/변경
├── usePartners.ts         # 파트너 데이터
├── useSMS.ts              # SMS 발송
├── useToast.ts            # 토스트 알림 (shadcn)
├── useConfirm.ts          # 확인 다이얼로그
├── useMediaQuery.ts       # 반응형 감지
└── useDebounce.ts         # 검색 디바운스
```

#### API 클라이언트 모듈화

```
lib/
├── api/
│   ├── client.ts          # Axios 인스턴스, 인터셉터
│   ├── applications.ts    # 신청 API
│   ├── partners.ts        # 파트너 API
│   ├── auth.ts            # 인증 API
│   └── index.ts           # 통합 export
└── utils/
    ├── format.ts          # 포맷팅 (날짜, 전화번호)
    ├── validation.ts      # 유효성 검사 스키마 (zod)
    └── constants.ts       # 상수 정의
```

### 10.1 개인정보 보호

| 항목        | 요구사항                               |
